# 009 — Автозапуск при входе через SMAppService

> Принято 2026-06-08. Реализовано: тумблер «Запускать при входе» в контекстном меню, сборка зелёная.

## Context

Нужно дать пользователю включать автозапуск Nook при входе в систему прямо из приложения, без ручной возни в Системных настройках. Заодно убрали бесполезный пункт «О Nook» (он только писал версию в лог, диалога нет) и вынесли версию наверх меню серым текстом — как принято в menu-bar приложениях.

## Decision

### Автозапуск — `SMAppService.mainApp`

Используем нативный `SMAppService` (macOS 13+, штатный путь на Tahoe) через крейт `objc2-service-management` (0.3.2 — та же линейка, что и остальные objc2-зависимости). Система сама регистрирует текущий `.app` как login item — **без** login-хелперов и LaunchAgent-плистов.

- `login.rs` — тонкий фасад над `unsafe` ServiceManagement: `is_enabled()` и `toggle()`. Контроллер не знает деталей API (Separation of Concerns).
- `is_enabled()` возвращает `true` при `Enabled` **и** при `RequiresApproval` — в обоих случаях пользователь намеренно включил автозапуск, галочка должна стоять (иначе тумблер «врёт»).
- `toggle()` — register/unregister, ошибку логирует как внешнюю нестабильность и не паникует.

### Меню

```
Nook 0.1.1            ← версия, disabled → система рисует серым
──────────────
Запускать при входе  ✓ ← тумблер, галочка по login::is_enabled()
──────────────
Выход            ⌘Q
```

Меню перестраивается при каждом правом клике (`menu::build`), поэтому галочка всегда свежая — состояние не кэшируем, читаем `status()` в момент построения.

## Связь с ADR 008 (КРИТИЧНО)

`mainAppService()` регистрирует путь к **тому `.app`, из которого запущен процесс**. Если включить автозапуск, когда Nook запущен из `target/debug`, в LoginItems пропишется dev-путь — ровно тот ad-hoc мусор, что засоряет `trackedApplications` в ControlCenter и ломает размещение якоря (`x=0`, см. ADR 008).

**Дисциплина:** автозапуск включать только из `/Applications/Nook.app`. В лог при `toggle()` пишем действие, чтобы при разборе было видно, откуда зарегистрировались. Код не блокирует включение из dev-пути жёстко — нельзя надёжно определить «правильную» установку, и пользователь волен ставить куда хочет; ограничение фиксируем дисциплиной и ADR.

## Alternatives

- **Login-хелпер (отдельный bundled helper + `SMLoginItemSetEnabled`).** Отвергнуто: устаревший путь до macOS 13, требует вложенного хелпер-бандла. `SMAppService.mainApp` проще и нативен для Tahoe.
- **LaunchAgent-плист вручную в `~/Library/LaunchAgents`.** Отвергнуто: обход штатного API, не виден в Системных настройках как login item, пользователь не может отключить через систему.
- **Два пункта меню (Включить/Выключить).** Отвергнуто: тумблер с галочкой чище и привычнее.

## Consequences

- Новая зависимость `objc2-service-management` (+ транзитивная `objc2-security`).
- Новый модуль `login.rs`; правки в `menu.rs`, `controller.rs`, `strings.rs`, `main.rs`.
- Пункт «О Nook» и метод `onAbout:` удалены; `menu_about` → `menu_login`.
- **Релиз / CI:** при подписи Developer ID (ADR 008) трекинг идёт по bundle id, и привязка автозапуска к пути перестаёт быть проблемой.
- `RequiresApproval` пользователь может встретить при первом включении — это нормально (macOS просит подтвердить в Настройках). Галочка при этом стоит.
