# 005 — UX-поведение: две зоны скрытия + жизненный цикл бара

**Статус: принято 2026-06-05.** Спроектировано эксперт-агентом после того, как gate показал сырое/непонятное поведение. Основа для фикса кода.

## Context

Технический фундамент (спейсер двигает иконки на Tahoe) заработал, но поведение было непонятным: спейсер прятал лишь 1-2 соседние иконки, не было ясной границы «что лишнее». Нужно было спроектировать цельный UX до доделки кода. Сырой кейс пользователя — `docs/drafts/ux-behavior-raw.md`.

## Decision

### 1. Модель скрытия — два разделителя, три зоны (как HiddenBar/Ice)

```
[ системные: время, wifi, Control Center ] [ ‹ видимый разделитель ] [ скрываемые иконки (зона B) ] [ невидимый спейсер ] [ всегда видны (зона A) ]
```

- **Зона A (всегда видны)** — правее спейсера.
- **Зона B (скрываемые)** — между `‹` и спейсером. Сюда пользователь Cmd+drag кладёт Яндекс.Диск, Docker, Ollama и т.д.
- **Зона C (системные)** — не трогаем.

Спейсер раздувается (`length=10000`) → зона B уезжает за левый край. `‹` остаётся виден (он правее спейсера). Визуальное правило для юзера: **«всё между `‹` и невидимой чертой — прячется»**. Пользователь сам управляет размером зоны B перетаскиванием.

<!-- DEPRECATED 2026-06-05: аксиома «создан первым → окажется левее» оказалась неверной/недетерминированной на Tahoe (ревизия кода, баг «клик прячет сам ‹»). Рабочий эталон nanobar делает НАОБОРОТ и закрепляет позиции через setAutosaveName. Корректное правило — ниже. -->

**Критично для кода (исправлено по ревизии):** порядок NSStatusItem в строке меню на Tahoe **не определяется** порядком вызова `statusItemWithLength` — он зависит от `setAutosaveName` и истории Cmd+drag пользователя. Поэтому:
- Обоим item (спейсеру и `‹`) задавать стабильный `setAutosaveName` (иначе порядок недетерминирован).
- Привести к доказанной схеме рабочего `nanobar`: **видимый `‹` создавать ПЕРВЫМ, невидимый спейсер — ВТОРЫМ; раздувать спейсер**. В nanobar при раздувании спейсера (созданного вторым) видимый элемент остаётся на экране.
- Эмпирически проверить фактический порядок (залогировать X-координаты `button.frame`/`window.frame.origin.x` обоих item на старте) и от него выбирать, какой item раздувать. Не полагаться на «первый = левее».

### 2. Связь спейсер ↔ бар-полоска — один жест, две функции синхронно

Клик `‹` для пользователя = «открыть бар». За кадром — два независимых события вместе:
1. спейсер раздувается → зона B скрывается;
2. NSPanel-полоска выезжает снизу со своими ярлыками.

Обратный жест (`‹` или таймер) сворачивает оба. Разделять на разные триггеры — лишняя путаница. Синхронизация = понятный UX.

### 3. Жизненный цикл бара (автомат)

Три достижимых состояния:
- `STATE_1: Shown × Closed` — базовое, иконки видны, бара нет.
- `STATE_2: Hidden × Open` — бар открыт, иконки за краем.
- `STATE_3: Hidden × ClosingPending` — бар открыт, таймер 5с тикает.

Переходы:
```
STATE_1 + клик_‹/hotkey        → раздуть спейсер + показать NSPanel → STATE_2
STATE_2 + клик_‹/hotkey        → скрыть NSPanel + сжать спейсер     → STATE_1
STATE_2 + mouse_exit/resignKey → запустить таймер 5с                → STATE_3
STATE_2 + клик_ярлыка          → NSWorkspace.open(app), не меняем   → STATE_2
STATE_3 + mouse_enter          → отменить таймер                    → STATE_2
STATE_3 + клик_‹/hotkey        → отменить таймер + закрыть           → STATE_1
STATE_3 + таймер_истёк         → закрыть                            → STATE_1
```
Клик вне бара (не по `‹`) = как mouse_exit (запускает ClosingPending, не закрывает сразу) — Bartender-style.

### 4. Отслеживание фокуса без хаков

- **NSTrackingArea** на contentView NSPanel (`mouseEnteredAndExited + activeAlways`): `mouseEntered` → отменить таймер (STATE_3→2), `mouseExited` → запустить таймер (STATE_2→3). Чистый AppKit, прав не нужно.
- **NSWindowDelegate `windowDidResignKey`** — доп. сигнал, если фокус ушёл в другое приложение.
- NSPanel с `.nonactivatingPanel` не становится key window — поэтому основной механизм именно NSTrackingArea.
- Таймер — `Timer`/`DispatchSourceTimer` на main thread; отменять при переходе в STATE_1/STATE_2.
- Пересоздавать NSTrackingArea при смене frame (`windowDidChangeScreen`/`viewDidLayout`).

### 5. Визуальный фидбек (MVP)

Смена символа: `‹` (иконки видны, бар закрыт) ↔ `›` (иконки скрыты, бар открыт). Для MVP этого достаточно. Счётчик скрытых иконок — нельзя честно (не знаем число чужих NSStatusItem), отложить. Анимация появления — после MVP.

Позиция бара: `originY = screen.maxY - menuBarHeight - barHeight`, ширина экрана, высота ~32-36pt.

## Alternatives

- **Одна граница (всё левее `<`)** — проще, но менее гибко (нет «всегда видимых»). Отклонено в пользу двух зон как у HiddenBar.
- **Спейсер и бар на разных триггерах** — отклонено: путаница «иконки скрыты, а бара нет».

## Consequences

- Фикс кода: спейсер регистрировать ПЕРВЫМ, `‹` вторым (исправляет сырое поведение gate). <!-- ОТМЕНЕНО ШАГОМ 1: см. раздел «Поправка по итогам ШАГА 1» ниже — на практике якорь создаётся первым, а надёжность даёт guard, а не порядок создания. -->
- Появляется автомат состояний — основа модулей `app.rs` (оркестрация) + `panel.rs` + `autohide.rs`.
- NSTrackingArea и таймер — часть спринтов 03/04.
- MVP-граница: два разделителя + спейсер + NSPanel + tracking + таймер 5с + смена символа. Без анимации/онбординга/мультимонитора (потом).

## Поправка по итогам ШАГА 1 (2026-06-05)

ШАГ 1 (спринты 01–02) реализован и проверен вживую на Tahoe. Часть проектных предположений этого ADR подтвердилась, одно — нет. Историю выше не трогаем, здесь — что выяснилось на практике.

**Подтвердилось:**
- Спейсер реально двигает иконки на Tahoe — раздувается до ~ширины экрана и выталкивает иконки левее себя за край. Якорь `‹` остаётся виден (он правее спейсера).
- Позиции айтемов надо закреплять `setAutosaveName` — иначе порядок плавает. Сделано для обоих айтемов.

**Не подтвердилось — главная поправка про порядок айтемов:**

Аксиома «создан раньше → окажется левее» (и обратная формулировка из раздела Decision §1) на Tahoe НЕ работает. Порядок `NSStatusItem` в строке меню НЕ определяется порядком вызова `statusItemWithLength` и не управляется программно — публичного API двигать свои или чужие айтемы НЕТ. Это подтверждено вживую (баг «клик прячет сам `‹`»), а также через Apple docs и чтение исходника HiddenBar (dwarvesf/hidden). Нативного API «скрыть чужую иконку» тоже не существует — только выталкивание шириной спейсера.

Поэтому надёжность даёт НЕ порядок создания, а **GUARD**: перед раздуванием спейсера контроллер сверяет реальные X-координаты обоих айтемов (`button().window().frame().origin.x`) и раздувает спейсер ТОЛЬКО если он строго левее якоря. Если нет — скрытие блокируется, якорь показывает `⚠<` и тултип просит поправить порядок Cmd+drag. Это убрало баг «`‹` улетел за край»: в опасной конфигурации мы просто ничего не делаем.

Фактический порядок создания в коде: якорь `<` — первым, спейсер — вторым (обратно §1 Decision). Но это деталь, а не гарантия: опираемся на guard, а не на порядок.

**Прочие факты ШАГА 1:**
- Координаты айтемов доступны не сразу, а ~0.5–1с после старта (после стабилизации RunLoop). Отсюда стартовое авто-скрытие через `NSTimer` 1с, а не мгновенно.
- Ширина скрытия ограничена (clamp 500..4000, ширина экрана + запас 200). Значение `10000` из §1 Decision на Tahoe даёт патологию layout — приём ограничения подсмотрен у HiddenBar.
- Спейсер на ШАГЕ 1 пока ВИДИМЫЙ (полоска `▏`, 12pt) — временно, для отладки порядка. Невидимым/`┊`-в-редактировании станет в рамках ADR 006.

Доказанный технический факт «нет нативного API скрытия чужих иконок, только push шириной + guard» вынесен отдельно в research journal `003-status-item-order-and-hide.md`.
