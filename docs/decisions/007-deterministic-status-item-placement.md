# 007 — Надёжное размещение статус-айтемов: событие + retry

## Context

На macOS 26 Tahoe размещение `NSStatusItem` в строке меню **асинхронно и недетерминированно**. После создания айтема его `button.window.frame.origin.x` равен 0 и лишь спустя момент становится реальной координатой — а иногда **залипает на 0 навсегда** (айтем не получил слот). Когда у приложения два айтема (якорь `<` и спейсер), один из них в ~50% запусков застревал на x=0.

Последствие бага: guard видел `spacer.x=0 < anchor.x=1702` как «спейсер левее», раздувал спейсер от нуля через весь экран и **уносил якорь** — `<` пропадал. Это и был симптом «то работает, то нет», «спейсер прячет всё».

Замеры (тестовый стенд, вердикт по координатам): чистая версия давала **40% успешных запусков**. Прямой запуск бинаря без `.app` невозможен на Tahoe вообще (нет identity → координаты 0/0); штатный запуск всегда через `.app` + `open`.

## Decision

Размещение делаем **бронебойным в три слоя**:

1. **Порядок создания.** Спейсер создаётся **первым** (fixed width стабильнее получает правый слот), якорь (`variableLength`) — вторым.

2. **Событие как основной триггер.** Подписываемся на `NSWindowDidMoveNotification`. Система постит её, когда айтем получает реальную координату (`x: 0 → 1700`). Это **детерминированный сигнал размещения** — реагируем на точное событие, а не опрашиваем координату по таймеру. Когда оба айтема валидны (x>0), запускаем стартовое авто-скрытие и отписываемся.

3. **Fallback-таймер с retry и эскалацией.** Если айтем залип на x=0 без события — таймер пересоздаёт его (`removeStatusItem` + создать заново). Если одиночное пересоздание не помогает несколько раз подряд (новый садится на тот же 0) — **эскалация**: пересоздаём оба айтема (`recreate_both`), система раскладывает слоты с нуля. Потолок попыток защищает от вечного таймера.

Плюс **ужесточённый guard**: перед раздуванием требует `spacer.x > 0 && anchor.x > 0 && spacer.x < anchor.x`. Координата 0 = «не размещён», а не «левее» — безопасный отказ (якорь не уезжает).

Результат: **40% → 100%** размещений (25/25, ранее 40/40 на промежуточных итерациях).

## Alternatives

- **Оба айтема фиксированной ширины** (`NSSquareStatusItemLength` якорю). Отвергнуто: якорь вставал левее спейсера (порядок инвертировался), ломая механику раздувания. Якорю нужен `variableLength`.
- **Синхронное ожидание размещения** (spin RunLoop / `thread::sleep` в `applicationDidFinishLaunching`). Отвергнуто: блокировка главного потока **ломает само размещение** — айтемы вообще не встают (0–8% успеха). Размещение требует свободного главного RunLoop; работать с координатами можно только после возврата из обработчика запуска.
- **Только таймер-поллинг без события** (промежуточная версия). Работало (40/40), но реагировало на опрос, а не на факт. Событие точнее и быстрее; таймер оставлен как страховка.
- **Фиксированная задержка перед авто-скрытием** (`STARTUP_HIDE_DELAY=1с`, как было в v0.1.0). Отвергнуто: 1с иногда меньше времени размещения → toggle по невалидным координатам. Заменено привязкой к факту «оба размещены».

## Consequences

- Размещение детерминированно: ловим точное событие, retry — страховка от залипания. «Зависимость от случая» снята.
- Запуск только через `.app` (с `CFBundleIdentifier`) — на Tahoe это условие получения координат статус-айтемом, не только autosave. Прямой бинарь не поддерживается (и никогда не был штатным — `ctl.sh` всегда делал `open`).
- В коде появились `make_spacer`/`make_anchor`/`recreate_*`, placement-таймер и подписка на нотификацию — сложнее, чем наивное создание, но это плата за надёжность на Tahoe.
- Диагностический журнал (`crate::log`) фиксирует ход размещения (`placement готово [event]`, `retry:`, `guard:`) — полезно для разбора залипаний на проде.
- Подробности исследования и итераций: `docs/research/journals/004–007`. Тестовый стенд: `scripts/test-stand.sh`.
