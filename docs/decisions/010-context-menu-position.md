# 010 — Позиция показа контекстного меню: якорить к нижней кромке кнопки

> Принято 2026-06-08. Проверено: меню падает вниз без scroll-arrow, юзер подтвердил визуально.

## Context

Контекстное меню статус-айтема (правый клик по якорю `<`) мы показываем **вручную** через `popUpMenuPositioningItem_atLocation_inView`, а не назначаем `statusItem.menu`. Так левый клик остаётся свободным под toggle скрытия иконок, а меню вызывается только правым кликом.

Симптом: при **первом** открытии меню налезало верхом на строку меню (menu bar), и macOS вместо первого пункта рисовал scroll-up-arrow `^`. При наведении мышкой меню «доезжало» на место. Выглядело как недорисованное меню.

Перебрали несколько ложных гипотез — ни одна не была причиной:

- **Disabled-первый-пункт рисуется как scroll-arrow.** Меняли первый пункт с disabled на enabled — не помогло.
- **Неготовый layout меню.** Вызывали `menu.size()` перед показом, чтобы «прогреть» раскладку — не помогло.
- **Серый цвет версии через `attributedTitle`.** Подозревали, что кастомный атрибутированный заголовок ломает раскладку. Не при чём — откатили вместе с лишними Cargo-фичами (`NSColor`, `NSAttributedString`, `NSDictionary`).

## Decision

Настоящий корень — точка `origin`, которую мы передаём в `popUpMenuPositioningItem_atLocation_inView`.

Было `origin = (0, button.bounds.height + 4.0)` — точка **над** кнопкой. `NSStatusBarButton` не flipped, ось Y растёт вверх, поэтому `bounds.height + 4` — это выше верхней кромки. Меню вставало в эту точку своим верхним-левым углом → уходило вверх, налезало на menu bar → macOS показывал scroll-arrow `^`.

Фикс: `origin = (0.0, 0.0)` — **нижняя кромка** кнопки. Меню падает вниз, как у нативных status item. Файл: `src/controller.rs`, функция `show_menu`.

Канонический совет AppKit: для `NSStatusBarButton` меню якорят к нижней кромке кнопки — точка `(midX, 0)` в координатах кнопки (не flipped). Меню раскрывается вниз от строки меню, как и положено.

## Alternatives

- **Назначить `statusItem.menu = menu`.** Самый штатный способ показать меню status item — система сама позиционирует его правильно. Отвергнуто: тогда **левый** клик тоже открывает меню, а нам левый клик нужен под toggle скрытия иконок. Ручной показ через `popUpMenuPositioningItem` — плата за разделение левого и правого клика.
- **`menu.size()` / enabled-первый-пункт.** Лечили симптом (scroll-arrow), а не причину (позиция показа). Не работали — см. ложные гипотезы выше.

## Consequences

- Меню всегда раскрывается вниз с первого раза, scroll-arrow `^` больше нет.
- Ручной показ через `popUpMenuPositioningItem` сохранён — это условие того, что левый клик делает toggle, а не открывает меню.
- **Связь с ADR 007.** Не путать. 007 — про **placement самого статус-айтема** в строке меню (координата `x` залипала на 0, айтем не получал слот, scroll/исчезновение из-за раздувания спейсера). Здесь — про **точку показа меню** относительно уже размещённой кнопки. Разные баги, разные слои: 007 — где стоит айтем, 010 — куда падает его меню.
