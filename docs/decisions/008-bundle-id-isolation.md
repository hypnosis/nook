# 008 — Стабильный путь запуска и ControlCenter adhoc-binary tracking (почему статус-айтем был невидим)

> Принято 2026-06-08. Проверено: якорь `<` появился под родным bundle id `com.hypnosis.nook`, юзер подтвердил.

## Context

`NSStatusItem` (якорь `<` + спейсер) **не появлялся в строке меню** — координаты залипали на `x=0` — но **только** под bundle id `com.hypnosis.nook`. Тот же самый бинарь, запущенный под любым другим id, размещался нормально (`x=1299`).

Долгое расследование (journals 004–009) перебрало кучу ложных диагнозов: notch, баг размещения на Tahoe, зомби-регистрации в LaunchServices, отравленный autosave-домен. Все они мимо. Настоящий корень оказался в **ControlCenter** — системном демоне, который ведёт реестр menu-bar приложений.

## Decision

Реальная первопричина: macOS Tahoe ControlCenter держит реестр menu-bar приложений в

```
~/Library/Group Containers/group.com.apple.controlcenter/Library/Preferences/group.com.apple.controlcenter.plist
```

(ключ `trackedApplications`). Приложение, подписанное **ad-hoc** (`codesign -s -`, без Apple Team ID), трекается там **не по bundle id, а по пути к исполняемому бинарю** — запись вида `adhocBinary → file://.../target/debug/nook`.

За dev-цикл накопились конфликтующие записи от разных путей одного и того же id: `target/debug`, `target/release`, старый `com.danila.clearbar`. Система видела «несколько nook» (в Настройках всплывали два ghost-тумблера) и не отдавала чистый слот в строке меню → `x=0`. А свежий id (`the-nook`) попадал в нормальный bundle-режим, потому что грязных adhoc-записей по его пути ещё не было — поэтому и «работал».

Решение из трёх частей:

1. **Запускать `.app` только из стабильного пути** (`/Applications/Nook.app`), а не из `target/debug`. Тогда ControlCenter создаёт одну чистую запись, и драки за слот нет.
2. **Идеал для релиза — подпись Developer ID** (с Apple Team ID). Тогда трекинг идёт по bundle id, а не по пути, и проблема исчезает в принципе.
3. **В dev-цикле — чистка `trackedApplications`** от adhocBinary-записей nook (python `plistlib`, с бэкапом), когда накопились конфликты.

Проверено: `x=1056`, якорь `<` виден под родным `com.hypnosis.nook`.

## Alternatives

- **Сменить bundle id на чистый (`com.hypnosis.the-nook`).** Так и думали раньше — это была ошибочная гипотеза прошлой версии этого ADR. Отвергнуто: лечит симптом, а не причину. Новый id «работает» лишь пока по его пути нет грязных adhoc-записей; первая же сборка из `target/debug` снова всё отравит. Плюс смена id ломает существующих пользователей. Корень — не id, а ad-hoc трекинг по пути.
- **Чинить только код размещения (ADR 007).** Недостаточно: 007 поднял надёжность кодом (событие + retry), но при грязном `trackedApplications` система всё равно не отдавала слот.

## Consequences

- **Bundle id `com.hypnosis.nook` сохранён** — не ломаем существующих пользователей.
- **Размещение надёжно при запуске из стабильного пути** — ControlCenter держит одну чистую запись вместо конфликтующего набора.
- **Обязательство по разработке:** ad-hoc сборки из меняющихся путей засоряют ControlCenter. Нужна дисциплина — запускать из `/Applications` или подписывать Developer ID. Иначе adhocBinary-записи снова накопятся.
- **Обязательство по релизу / CI:** добавить шаг подписи Team ID (Developer ID) — тогда трекинг по bundle id и проблема снимается в корне.
- **Связь с журналами 004–008.** Этот ADR заменяет/уточняет их гипотезу о «недетерминированном `x=0`»: реальная причина была не в асинхронности API размещения, а в грязном `trackedApplications` от ad-hoc сборок.
- **Связь с ADR 007.** Дополняет 007: 007 чинил **код** (детерминированное размещение через событие + retry), 008 объясняет **окружение** (ControlCenter трекает ad-hoc бинарь по пути). Настоящий корень бага «то работает, то нет» был именно в окружении.
- Подробности расследования: `docs/research/journals/004–009`.
