# 009 — ControlCenter трекает наш бинарь по пути (ad-hoc подпись): якорь не появлялся

**Дата:** 2026-06-08
**Статус:** ОКОНЧАТЕЛЬНЫЙ КОРЕНЬ + РЕШЕНИЕ (проверено визуально). Якорь `<` снова появился в menu bar под РОДНЫМ id `com.hypnosis.nook`, юзер подтвердил.

Короткий итог: айтемы не вставали в видимую зону не из-за notch, не из-за бага Tahoe и не из-за зомби LaunchServices. Виноват реестр ControlCenter: из-за ad-hoc подписи (без Apple Team ID) macOS Tahoe трекала наше приложение не по bundle id, а по ПУТИ к бинарю. За время разработки накопились конфликтующие записи от разных путей одного id (`target/debug`, `target/release`, осколки старого имени). Несколько записей на один id = система видит «несколько nook» и не отдаёт чистый слот → координата залипает на `x=0`. Лечится чисткой этого реестра и запуском из одного стабильного пути.

## Симптом
Якорь `<` и спейсер не появлялись в menu bar. Координаты намертво залипали на `x=0` — вечный retry, потом «лимит исчерпан, сдаюсь».

Ключевое наблюдение: это происходило ТОЛЬКО у bundle id `com.hypnosis.nook`. Тот же самый бинарь, запущенный с другим id (`com.hypnosis.the-nook`), сразу вставал на `x=1299`. Причём раньше `com.hypnosis.nook` на этой машине работал нормально.

## Ложные диагнозы (отброшены по пути)
- **notch-клиппинг (journal 008)** — опровергнут скриншотом юзера: до notch места в баре полно, свободная зона есть.
- **баг Tahoe, NSStatusItem не рисуется** — нет, другие сторонние приложения свои айтемы показывают, система чужие NSStatusItem рисует.
- **зомби-регистрации `/Volumes/nook` в LaunchServices** — РЕАЛЬНО были (три disk-image контейнера от удалённых тестовых DMG), вычищены через `lsregister -delete` + перезагрузка. Флаги `-u` / `-gc` / `-kill` не сработали: `-u` на несуществующих путях даёт ошибку `-10814`, `-kill` Apple удалила. НО даже после полной чистки LaunchServices и ребута `x=0` остался — значит зомби LaunchServices НЕ были корнем (или не единственным).
- **отравленный autosave-домен** — опровергнут: домены и `com.hypnosis.nook`, и рабочего `com.hypnosis.the-nook` оба ПУСТЫ. Позиция вообще не сохранялась, autosave ни при чём.
- **CrashReporter Intervals троттлинг** — проверен (нашлась запись со счётчиком 3183), удаление не помогло.

## Настоящий корень (подтверждён)
macOS Tahoe ControlCenter ведёт собственный реестр menu-bar приложений. Лежит он здесь:

```
~/Library/Group Containers/group.com.apple.controlcenter/Library/Preferences/group.com.apple.controlcenter.plist
```

Внутри — ключ `trackedApplications`: это вложенный binary plist (список словарей, по словарю на приложение).

Нормальное приложение трекается по BUNDLE ID:

```python
{'bundle': {'_0': 'com.hypnosis.the-nook'}}
```

А наш `com.hypnosis.nook` трекался по ПУТИ К БИНАРЮ:

```python
{'adhocBinary': {'_0': {'relative': 'file:///Users/hypnosis/projects/nook/target/debug/nook'}}}
```

Причина — ad-hoc подпись. Бинарь подписан `codesign -s -`, у него `TeamIdentifier=not set`. Без Team ID ControlCenter не доверяет bundle id как стабильному идентификатору и трекает приложение по пути к исполняемому файлу.

За время разработки накопились конфликтующие `adhocBinary`-записи от разных путей одного и того же id: `target/debug/nook`, `target/release/nook`, плюс осколок старого `com.danila.clearbar`, а сам `com.hypnosis.nook` ещё и застрял внутри чужой записи (ghostty). Несколько `adhocBinary`-записей на один bundle id = система видит «несколько nook» (в Настройках → Строка меню появляются два серых ghost-тумблера) и не даёт чистый слот → `x=0`.

## Почему `the-nook` работал
Свежий, ни разу не использованный id. ControlCenter записал его в НОРМАЛЬНОМ bundle-режиме — одной чистой записью `{'bundle': {'_0': 'com.hypnosis.the-nook'}}`, а не через `adhocBinary`. Конфликтов нет → чистый слот → `x=1299` с первой попытки.

То есть дело было не в имени `nook` как таковом, а в том, что его реестровая запись была загрязнена путями.

## Решение (3 шага, сработало — `x=1056`, юзер подтвердил)
1. **Вычистить `trackedApplications`.** Удалить все `adhocBinary`-записи nook (фильтр по подстроке `projects/nook`), а также осколки `com.hypnosis.nook` / `com.danila.clearbar`, застрявшие в чужих `menuItemLocations`. Делается через python `plistlib`: распаковать вложенный binary plist, отфильтровать список, записать обратно. **Бэкап файла обязателен.**
2. **Запускать `.app` из стабильного пути** `/Applications/Nook.app`, а НЕ из `target/debug/`. Тогда ControlCenter создаёт одну чистую запись по фиксированному пути.
3. **Перезапустить демоны:** `killall cfprefsd ControlCenter`.

Результат: `x=1056` / `x=1082` (видимая зона), якорь `<` появился под РОДНЫМ id `com.hypnosis.nook` — без всякой смены имени.

## Урок
- Ad-hoc подпись (без Apple Developer Team ID) заставляет ControlCenter на Tahoe трекать menu-bar приложение по ПУТИ к бинарю, а не по bundle id. Запуск одного id из разных путей (debug / release / перемещение `.app`) плодит конфликтующие записи → невидимый статус-айтем.
- Для стабильного поведения есть три варианта: подписывать Developer ID (есть Team ID → трекинг по bundle id), либо ВСЕГДА запускать из одного фиксированного пути (`/Applications`), либо периодически чистить `trackedApplications` в dev-цикле.
- Это объясняет всю сагу journals 004–008: «недетерминированное размещение на `x=0`» было следствием грязного `trackedApplications` от итеративных ad-hoc сборок, а не асинхронности AppKit API.
