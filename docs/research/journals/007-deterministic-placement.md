# 007 — Бронебойное размещение: события вместо поллинга

**Дата:** 2026-06-07
**Вопрос:** уйти от retry «на удачу» к детерминизму — знать СОБЫТИЕ размещения NSStatusItem, а не пересоздавать вслепую.
**Источники:** context7 (objc2-app-kit docs), perplexity (Apple AppKit docs).

## Главный факт
**Детерминированного события «NSStatusItem размещён, frame финальный» в публичном AppKit НЕТ.**
- Нет KVO-свойства «layout complete» на NSStatusItem / button / button.window.
- Нет нотификации «placement complete».
- `applicationDidFinishLaunching` НЕ гарантирует, что координаты финальны.
Подтверждено обоими источниками (Apple docs developer.apple.com/documentation/AppKit/NSStatusItem).

## Что РЕАЛЬНО доступно (точнее поллинга)

### 1. NSWindowDidMoveNotification (есть в objc2-app-kit ✅)
Окно постит её при изменении frame. Окно кнопки статус-айтема получает её, когда система
двигает айтем на финальную позицию. → Подписаться на неё для `button.window` и реагировать
на РЕАЛЬНОЕ событие перемещения вместо таймера-поллинга.
Также есть NSWindowDidResizeNotification.

### 2. Канонический паттерн Apple
- создать оба айтема рано, держать strong-ссылки;
- НЕ зависеть от мгновенных координат после create;
- отложить геометро-зависимую работу до завершения первого layout-прохода RunLoop
  (следующий тик main queue), потом проверить frame.

### 3. Природа x=0 (из доков)
`button.window.frame.origin.x` отражает окно НА МОМЕНТ СОЗДАНИЯ, до того как AppKit
позиционировал menu-bar-extra. Это timing/lifecycle, не «сигнал слота». Надёжно — дождаться
пока окно онскрин и frame ненулевой+стабильный.

## Развилка: как сделать бронебойнее текущего retry

**Текущее (работает 40/40, но «по таймеру»):** placement-таймер 0.3с поллит x, пересоздаёт
застрявших, эскалация recreate_both. Минус — реагирует не на событие, а на опрос.

**Вариант A — подписка на NSWindowDidMoveNotification (событийно):**
Подписаться на нотификацию для окна каждого айтема. При срабатывании проверить frame;
когда оба валидны — действовать. Плюс: реакция на реальное событие размещения, не поллинг.
Минус: окно айтема может не постить её при ПЕРВОМ размещении (только при ДВИЖЕНИИ) — надо
проверить экспериментально. Если первое размещение не шлёт нотификацию — событие не придёт.

**Вариант B — таймер как сейчас, но обоснованный (defensive):**
Оставить таймер, т.к. детерминированного события НЕТ по докам. Таймер + retry + эскалация —
это и есть «канонический паттерн Apple» (отложить до layout-прохода, перепроверить frame).
Плюс: уже 40/40. Минус: не событийный.

**Вариант C — гибрид:** подписка на NSWindowDidMoveNotification КАК ОСНОВНОЙ триггер +
таймер как fallback (если событие не пришло за N мс). Бронебойнее всего: событие когда есть,
страховка когда нет.

## ЭКСПЕРИМЕНТ ПРОВЕДЁН → СОБЫТИЕ СУЩЕСТВУЕТ ✅✅

Подписка на NSWindowDidMoveNotification (глобально) ловит размещение В РЕАЛЬНОМ ВРЕМЕНИ.
Лог 3 запусков показал точную эволюцию координат через событие:
```
EVENT WindowDidMove: spacer=0    anchor=0     ← создание
EVENT WindowDidMove: spacer=1700 anchor=0     ← спейсер РАЗМЕЩЁН (событие!)
EVENT WindowDidMove: spacer=1700 anchor=1702  ← якорь РАЗМЕЩЁН (событие!)
```

**Вывод:** perplexity ошибся («не guaranteed»). На практике (Tahoe) событие приходит НАДЁЖНО
при каждом изменении frame айтема, включая первое размещение x:0→реальная координата.
Это и есть детерминированный сигнал, которого «нет в доках», но он работает.

## РЕШЕНИЕ: Вариант C — событие как основной триггер + таймер как fallback

Переход с поллинга на событийную модель:
- Подписка на NSWindowDidMove → на каждом событии проверяем оба айтема → когда оба валидны
  (x>0), запускаем стартовое авто-скрытие и ОТПИСЫВАЕМСЯ. Реакция мгновенная, без поллинга.
- Таймер-fallback оставляем: если айтем ЗАЛИП на 0 без события (редко) — retry/эскалация
  его пересоздаёт. recreate_* сам генерирует новое окно → новое событие.

Это бронебойно: ловим точное событие размещения (не гадаем по таймеру), а retry-таймер —
страховка от залипания. «Зависимость от случая» снята: есть событие + есть fallback.
