# 004 — Якорь `<` не появляется: гипотезы про bundle identity и координаты

**Дата:** 2026-06-07
**Статус:** ЗАКРЫТО. Все гипотезы проверены замерами. Решение → ADR 007.
**Контекст:** после фичи автозапуск (SMAppService) и чистки LaunchServices даже стабильная v0.1.1 перестала показывать `<` при прямом запуске бинаря.

## Итоговые вердикты по гипотезам
| Гип. | Суть | Вердикт |
|------|------|---------|
| H1 | bare-бинарь без `.app` не получает координат на Tahoe | ✅ подтверждена (нужен `.app`; «раньше без .app» — миф, ctl.sh всегда `open`) |
| H2 | тайминг: координаты дозревают со временем | ❌ опровергнута (x=0 залипает навсегда, poll 5с) |
| H3 | спейсер на x=0 раздувается и прячет всё | ✅ подтверждена (следствие невалидных координат) |
| H4 | **ГЛАВНЫЙ:** размещение недетерминированно, guard обманут нулём | ✅✅ подтверждена и воспроизведена (40% baseline) |
| H5 | автозапуск/SMAppService ломает запуск | ❌ опровергнута (виноваты DIAG-правки в том же stash) |

**Решение и фикс:** см. [ADR 007](../../decisions/007-deterministic-status-item-placement.md) и итерации в [journal 006](006-fix-iterations-plan.md). Детерминированное событие размещения — [journal 007](007-deterministic-placement.md). Прогресс 40%→100%.

## Что наблюдаем (факты по координатам)

Один и тот же бинарь v0.1.1 (код чистый, `git diff v0.1.1 == пусто`), два режима запуска:

| Режим запуска | spacer.x | anchor.x | guard | результат |
|---------------|----------|----------|-------|-----------|
| Прямой бинарь (`./target/release/nook`) | **0** | **0** | `0<0` = false → СТОП | айтемы невидимы, скрытие заблокировано |
| Из `.app` (`open Nook.app`) | **1700** | **1702** | `1700<1702` = true → ✓ | скрытие работает, спейсер раздулся до 1928, якорь на месте |

Canonical-чтение координат (`convertRectToScreen`) для прямого бинаря:
`spacer.screen=8, anchor.screen=0` — это **левый край экрана** (под меню Apple), а не правый кластер иконок (~1700). То есть айтемы физически свалены в левый угол.

`bundleID` прямого процесса = **NULL** (проверено `lsappinfo info`).
Домен `defaults read com.hypnosis.nook` = **пуст** → autosave позиций НИКОГДА не сохранялся.

## Гипотезы

### H1 — Bare-бинарь не получает координат статус-айтема на Tahoe ✅ (ПОДТВЕРЖДЕНА, уточнена)

**Формулировка (уточнённая):** на macOS 26 Tahoe `NSStatusItem` получает реальное место в menu bar только если процесс запущен как **bundled application** (`fileType=APPL`). Это достигается ТОЛЬКО реальным `.app` + LaunchServices (`open`). Ни строка bundle id, ни регистрация в реестре сами по себе не помогают.

**Доказательство (evidence-таблица процессов):**

| | Прямой бинарь | Из .app |
|---|---|---|
| `bundleID` | NULL | com.hypnosis.nook |
| `fileType` | **????** | **APPL** ← ключевое |
| координаты | 0/0 ❌ | 1700/1702 ✅ |

**Опровергнутые обходы:**
- **T1 — lsregister -f .app → прямой бинарь = 0/0.** Регистрация `.app` в LaunchServices НЕ даёт identity прямому процессу. ⇒ гипотеза «раньше работало из-за призраков в реестре» **ОПРОВЕРГНУТА**.
- **T2 — `__CFBundleIdentifier=...` env → прямой бинарь = 0/0, bundleID=NULL.** Env-переменная не делает процесс приложением.

**Вывод:** различие не в bundle id как строке, а в `fileType=APPL` — статусе «настоящее приложение», который выдаётся только через `.app`-обёртку. **На Tahoe `.app` обязателен.**

**ОТКРЫТЫЙ ВОПРОС — ЗАКРЫТ ✅ (T-history):** «раньше работало без .app» — **посылка неверна**. Штатный скрипт запуска `ctl.sh` всегда делал `open "$APP_PATH"` где `APP_PATH=nook.app` (ctl.sh:54). То есть «раньше» ВСЕГДА запускали через `.app` + `open`, прямой бинарь `./target/release/nook` никогда не был штатным способом. Координаты 1185/1213 из истории — это запуск из `.app`, не прямой. Загадки нет: прямой бинарь давал 0/0 всегда, просто им не пользовались.

**Финальный статус H1:** на Tahoe статус-айтем требует `fileType=APPL` (реальный `.app` + `open`). Прямой бинарь невозможен — и никогда не использовался. Решение запуска = `.app`, как и было в `ctl.sh`.

### H2 — Раннее раздувание спейсера ломает координаты (тайминг) ⚠️ (частично, под вопрос)

**Формулировка:** стартовый авто-скрытие через 1с срабатывает ДО стабилизации координат. В этот момент координаты ещё 0/0 → спейсер раздувается на позиции 0 → растёт через весь экран → перекрывает якорь.

**Наблюдение:** в `.app`-режиме без раннего toggle координаты валидны (1700/1702). С toggle через 2с — якорь НЕ уехал (остался 1702). Значит при стабильных координатах раздувание безопасно. Под вопрос: что происходит при toggle ровно на 1с (как в проде STARTUP_HIDE_DELAY).

**Тест:** T3 — варьировать STARTUP_HIDE_DELAY (0.5 / 1 / 2 / 3с), логировать координаты на момент toggle.

---

### H3 — «Spacer прятал всё, пока guard не написали» — это режим x=0

**Формулировка:** без guard'а спейсер на x=0 раздувается на 1928 → растёт вправо через весь экран → прячет/перекрывает ВСЁ, включая якорь. Guard (`spacer.x < anchor.x`) это останавливает. При x=0/0 guard честно блокирует → видно «ничего не происходит», а не «всё спряталось».

**Связь:** H3 = следствие H1. Корень один — невалидные координаты при отсутствии identity.

**Тест:** T4 — прямой бинарь без guard (закомментировать guard), снять что с раскладкой при раздувании на x=0.

### H4 — ГЛАВНЫЙ БАГ: координаты недетерминированы при старте, guard обманывается нулём ✅✅ (ПОДТВЕРЖДЕНА, ВОСПРОИЗВЕДЕНА)

**Формулировка:** autosave пуст → айтемы встают непредсказуемо. Стартовый таймер (1с) ловит координаты в случайный момент. Guard `spacer.x < anchor.x` пропускает случай `spacer.x=0` (левый край), но раздувание спейсера на 1928 ОТ нуля растёт вправо через весь экран → перекрывает всё, включая якорь.

**Доказательство — 5 запусков чистой v0.1.1 из .app, ТРИ разных исхода:**

| Запуск | spacer.x | anchor.x | исход |
|--------|----------|----------|-------|
| 1,3,4 | 1700 | 1702 | ✅ норма |
| 2 | **0** | 1702 | ❌ guard обманут, раздувание от 0 ломает раскладку |
| 5 | **0** | **0** | ❌ координаты не готовы, guard блокирует |

**~40% запусков ломаются.** Это и есть «то работает, то нет», «спейсер прячет всё», «`<` уехал». Корень — НЕ bundle/identity (это отдельная H1, для прямого бинаря), а **недетерминированная расстановка айтемов + наивный guard, доверяющий нулю**.

**Два дефекта guard:**
1. Принимает `spacer.x == 0` как валидное «левее» — но 0 это «не размещён / левый край», а не «слева от иконок».
2. Срабатывает по фиксированному таймеру (1с), не дожидаясь стабилизации координат.

**Направление фикса (НЕ реализовывать без юзера):**
- Guard должен отвергать `spacer.x == 0` и `anchor.x == 0` (невалидно).
- Guard должен требовать осмысленный зазор: спейсер левее якоря НА ШИРИНУ ИКОНОК, не на 2px.
- Вместо фикс-таймера — поллинг координат до стабилизации (condition-based waiting), потом действие.
- Решить судьбу autosave: либо заставить работать (сохранять/грузить позицию), либо не полагаться на него и вычислять порядок программно.

**Тесты T5 (autosave после Cmd+drag), T6 (порядок создания) — для уточнения, нужен юзер для T5.**

### H5 — SMAppService (автозапуск) ломает запуск ❌ (СКОРЕЕ ВСЕГО НЕВЕРНА)

**Формулировка:** фича автозапуск через SMAppService ломает координаты/запуск.

**Анализ stash (фича автозапуск, `git stash show -p`):** автозапуск НЕ трогает координаты/спейсер/guard. Он добавляет:
- `Cargo.toml`: зависимость `objc2-service-management`
- `menu.rs`/`strings.rs`: пункт меню «Автозапуск» (галочка) вместо «О Nook»
- `controller.rs`: `onToggleAutostart:` (вкл/выкл Login Items)
- `src/autostart.rs`: регистрация SMAppService

**НАСТОЯЩАЯ причина «поломки» — диагностические правки в ТОМ ЖЕ stash:**
1. **`status_bar.rs`: `spacer.setVisible(false)`** — спейсер СКРЫТ «для диагностики». Без спейсера механика скрытия не работает в принципе.
2. **`controller.rs`:** стартовое авто-скрытие ЗАКОММЕНТИРОВАНО, заменено DIAG-таймером 0.5с.
3. **`ctl.sh`:** `nook.app` → `Nook.app` (смена регистра).

**Вывод:** «сломалось после автозапуска» — иллюзия совпадения. В одной сессии добавили автозапуск И нагородили DIAG-правок (скрытый спейсер, отключённое автоскрытие). Сломали именно DIAG-правки, не автозапуск. **Стабильная v0.1.1 (без stash) из `.app` работает — проверено (baseline 1700/1702, скрытие отрабатывает).**

**Остаточная проверка (T7, нужен юзер):** запуск по логину через SMAppService vs `open` — снять координаты, убедиться что launchd-контекст не ломает (низкий приоритет, гипотеза уже опровергнута анализом).

## Сводный вывод (ФИНАЛ, по результатам тестов)

**Два независимых корня, оба не про автозапуск:**

### Корень 1 — identity (H1): прямой бинарь невозможен на Tahoe
- Прямой бинарь = `fileType=????`, `bundleID=NULL` → координаты 0/0. Опровергнуты обходы: lsregister (T1), env (T2).
- «Раньше без .app» — миф: `ctl.sh` всегда делал `open Nook.app`. Прямой запуск не был штатным.
- **Запуск только через `.app`** (как и было).

### Корень 2 — недетерминизм спейсера (H4): ГЛАВНЫЙ баг, ломает даже стабильную v0.1.1
- Из `.app` **anchor стабильно 1702**, но **spacer в ~50% запусков застревает на 0**.
- **Поведение бинарное и НЕОБРАТИМОЕ во времени:** спейсер фиксируется при создании либо на 1700 (норма), либо на 0 (баг) — и НЕ меняется (проверено poll 6с, первый замер = последний).
- **Ожидание НЕ помогает** (condition-based waiting бесполезен — координата не «дозревает»).
- Guard обманывается: `spacer.x=0 < anchor.x=1702` проходит → раздувание на 1928 от нуля → перекрывает весь бар, якорь уезжает.

**Статистика:** 10 запусков → 5 OK (1700/1702), 5 BAD (0/1702). 50/50. Вот «то работает, то нет».

### Что НЕ виновато
- ❌ Автозапуск/SMAppService — чистый, не трогает координаты (H5 опровергнута анализом stash).
- ❌ Тайминг/ожидание — координата необратима (H2 опровергнута poll-тестом).
- ❌ LaunchServices-призраки — не дают identity прямому бинарю (T1).

### 🎯 ФИКС НАЙДЕН (T6): создавать СПЕЙСЕР ПЕРВЫМ

**Тест T6 — порядок создания решает недетерминизм:**

| Порядок | spacer.x | статистика |
|---------|----------|-----------|
| anchor первым (текущий код v0.1.1) | 0 или 1700 | **50/50 ❌** (5 BAD из 10) |
| **спейсер первым** (env NOOK_SPACER_FIRST=1) | 1700 всегда | **12/12 OK ✅** (0 BAD) |

**Почему:** macOS добавляет новые статус-айтемы слева от существующих. Когда спейсер (`fixed length`) создаётся первым, он стабильно получает слот справа, якорь (`variableLength`) встаёт рядом. Когда якорь `variableLength` создаётся первым — спейсер в ~50% не получает корректный слот, падает на x=0.

**Минимальный фикс (хирургический):** в `status_bar::create` поменять порядок — спейсер до якоря. Доказано env-флагом T6 на тестовой ветке fix/anchor-position. НЕ применён в код (юзер просил только исследование).

### Дополнительно — ужесточить guard (defense-in-depth)
Даже с фиксом порядка стоит добавить страховку в guard:
- отвергать `spacer.x == 0` и `anchor.x == 0` (невалидно);
- требовать осмысленный зазор (спейсер левее якоря, оба валидны).
Это защитит от редких краёв (другой дисплей, notch, ручной Cmd+drag).

### Осталось проверить (нужен юзер)
- T5 — autosave после ручного Cmd+drag (переживает ли перезапуск → можно ли стабилизировать позицию).
- T6 — спейсер первым vs якорь первым (детерминизм порядка).
- Визуальная проверка `<` глазами (CLI-скриншоты чёрные — нет Screen Recording).
