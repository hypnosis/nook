# 006 — План фикса по итерациям (с тестовым стендом)

**Дата:** 2026-06-07
**Ветка:** fix/anchor-position
**Метод:** 1 итерация = 1 гипотеза-фикс → прогон через тестовый стенд → вердикт по координатам.
**Вердикт:** статистика N запусков, считаем OK (spacer.x≈1700, anchor.x≈1700, spacer<anchor) vs BAD (любой x=0 или spacer не левее).

## Базовая линия (до фиксов, чистая v0.1.1)
Из `.app`: **40% OK** (4/10, стенд). spacer застревает на 0 в половине запусков.

---

## РЕЗУЛЬТАТЫ ИТЕРАЦИЙ

### ИТЕРАЦИЯ 1 — спейсер первым → ✅ ЧАСТИЧНО (40% → 58%), оставлена
- spacer теперь ВСЕГДА 1700 (починен!). Но проблема ПЕРЕЕХАЛА на anchor: теперь anchor=0 в ~40%.
- Вывод: «первый созданный стабилен, второй — нет». Подтверждает гипотезу асинхронности (итер.3).
- Решение: оставить (улучшила + починила спейсер), идти в итер.3 (бьёт корень — второй айтем не успевает).

### ИТЕРАЦИЯ 3 — синхронность через spin RunLoop → ❌ СЛОМАЛА (58% → 8%), ОТКАЧЕНА
- `wait_until_placed` крутил RunLoop внутри `applicationDidFinishLaunching` до создания второго айтема.
- Результат: 8% OK, появились NIL (айтемы вообще не размещаются).
- **Урок:** блокирующая прокрутка RunLoop в обработчике запуска ломает размещение айтемов. Spin-wait — тупик.
- Откачена, вернулись к итер.1 (58% подтверждено повторным прогоном).

### ИТЕРАЦИЯ 3b — отложенное создание второго айтема через таймер (асинхронно, не блокируя)
- Гипотеза: создать спейсер сразу, якорь — в коротком NSTimer (следующий тик RunLoop), чтобы система успела разместить спейсер БЕЗ блокировки обработчика.
- Статус: отложена (см. итер.2 — сначала понять природу anchor=0).

### ИТЕРАЦИЯ 2 — оба фикс. ширины (якорь square) → ❌ СЛОМАЛА (0%), ОТКАЧЕНА, но ВАЖНОЕ ОТКРЫТИЕ
- С квадратной шириной якорь встал на **1690 — ЛЕВЕЕ спейсера (1700)**. Порядок инвертировался.
- Вывод: оба фикс. ширины → второй созданный (якорь) встаёт СЛЕВА детерминированно. Для нашей механики (спейсер раздувается влево от якоря) нужен якорь СПРАВА → variableLength якорю нужен.
- Откачена.

### Природа anchor=0 — ВЫЯСНЕНА
- anchor=0 **навсегда** (poll 5с: первый замер = последний, не дозревает).
- immediate (синхронно после create) — **всегда 0**: координата присваивается асинхронно, ПОСЛЕ возврата из didFinishLaunching.

### ИТЕРАЦИЯ 3c — thread::sleep между создания → ❌ СЛОМАЛА (0%), ОТКАЧЕНА, КЛЮЧЕВОЙ ИНСАЙТ
- spacer=0 ВСЕГДА. Блокировка главного потока (sleep) мешает размещению.
- **ГЛАВНЫЙ ВЫВОД:** размещение айтема требует, чтобы ГЛАВНЫЙ ПОТОК крутил RunLoop. Любая блокировка (sleep ИЛИ вложенный spin RunLoop из итер.3) ломает размещение. Айтемы размещаются ТОЛЬКО после возврата из `applicationDidFinishLaunching`.
- Следствие: всё синхронное в `create()` не видит финальных координат. Корректное решение — АСИНХРОННОЕ, через таймеры после возврата из обработчика.

### ИТЕРАЦИЯ 4-retry (якорь) → ✅ ПОМОГЛО (58% → 78%)
- Таймер каждые 0.3с: если anchor.x=0 → removeStatusItem + создать заново (до 5 попыток).
- anchor теперь стабильно 1702 (retry чинит). Остаточные BAD (3/14) — теперь spacer=0.
- Подтверждение: проблема симметрична, ЛЮБОЙ айтем может застрять. retry якоря работает → распространить на спейсер.

### ИТЕРАЦИЯ 4b-retry (оба айтема) → ✅✅ РЕШЕНО (40% → 100%)
- retry для спейсера И якоря: если x=0 → removeStatusItem + создать заново (до 10 попыток, poll 0.3с).
- **Результат: 25/25 OK при паузе 4с (retry завершён).** На паузе 2.5с было 95-96% — артефакт замера (retry ещё шёл).
- ИТОГ фиксов: spacer первым (итер.1) + retry обоих (итер.4b) = детерминированные 1700/1702.

### ОСТАЁТСЯ — синхронизация toggle с retry
Стартовое автоскрытие на 1с может сработать ДО завершения retry. Надо: автоскрытие/первый toggle
запускать ТОЛЬКО после того, как retry-таймер подтвердил оба валидны (он уже это знает — в точке
«оба стабилизированы, poll остановлен»). Перенести запуск стартового toggle туда.

### ИТЕРАЦИЯ 5 — синхронизация toggle с retry → ✅ 95% (стенд)
- Убран отдельный стартовый таймер (1с). Первый toggle вызывается из placement-проверки в момент «оба размещены».
- Полный цикл в логе работает: spacer первым → anchor retry → оба ок → guard ✓ → скрытие.
- Остаточный BAD ~5% = guard поймал `0/0` (редкая гонка двух retry) и БЕЗОПАСНО ОТКАЗАЛ (якорь не уезжает, `<` виден, просто не спряталось). Не поломка бара.

### ИТЕРАЦИЯ 6 — ужесточить guard → ✅ 100% (20/20)
- guard теперь требует `spacer.x > 0 && anchor.x > 0 && spacer.x < anchor.x`. x=0 = «не размещён», не «левее».
- Все фиксы вместе (спейсер первым + retry обоих + синхронный toggle + ужесточённый guard) = 100%.

## ИТОГОВАЯ СВОДКА ФИКСОВ (применены в ветке fix/anchor-position)
1. **status_bar.rs:** спейсер создаётся первым; вынесены make_spacer/make_anchor; recreate_spacer/recreate_anchor для retry.
2. **controller.rs:** placement-таймер (0.3с) пересоздаёт застрявшие на x=0 айтемы (до 10 попыток каждый), при «оба размещены» запускает стартовое авто-скрытие и останавливается; ужесточённый guard (оба x>0).
3. Убран фиксированный STARTUP_HIDE_DELAY (toggle теперь по факту размещения, не по таймауту).

**Прогресс: 40% → 100%.** Тестстенд: `scripts/test-stand.sh N`.

### ИТЕРАЦИЯ 7 — эскалация recreate_both + защита от вечного таймера → ✅ 100% (20/20)
- Пойман редкий край (~3%): спейсер 1700, но якорь застревает на 0 ПОВТОРНО — каждое одиночное пересоздание якоря садится на тот же 0 (все 10 попыток впустую).
- Фикс: после PLACEMENT_ESCALATE_AFTER (4) безуспешных проверок — `recreate_both` (удалить оба, пересоздать композицию с нуля → система перераскладывает слоты).
- Заодно: PLACEMENT_MAX_ATTEMPTS (30) — стоп таймера, если размещение не удаётся вообще (чинит логдыру «вечный таймер при исчерпании попыток»).

### ИТЕРАЦИЯ 8 — событийная модель (бронебойность) → см. journal 007
- Найдено детерминированное СОБЫТИЕ размещения: NSWindowDidMoveNotification приходит при x:0→реальная координата (эксперимент в journal 007).
- Переход с поллинга на событие: подписка = ОСНОВНОЙ триггер старта авто-скрытия; таймер = FALLBACK с retry+эскалацией.
- Идемпотентность через placement_done; отписка от нотификации после финиша.
- Лог подтверждает «placement готово [event]» — стартует именно событие.

## Тестовый стенд
Скрипт `scripts/test-stand.sh` (создать):
- собирает release, заворачивает в `.app` (минимальный Info.plist, ad-hoc подпись)
- N запусков через `open`, между ними kill + пауза
- парсит `/tmp/nook-debug.log`, считает OK/BAD, печатает таблицу + итог
- НЕ требует Screen Recording — вердикт по координатам из лога

## Итерации (по приоритету)

### ИТЕРАЦИЯ 1 — порядок создания: спейсер ПЕРВЫМ (найденный фикс T6)
**Гипотеза:** спейсер (fixed) создаётся первым → стабильно получает слот → spacer.x всегда ≈1700.
**Правка:** `status_bar::create` — поменять порядок (спейсер до якоря).
**Доказано в T6:** 12/12 OK с env-флагом. Теперь зашить в код.
**Ожидание:** ≥ базы (50%), цель 100%.

### ИТЕРАЦИЯ 2 — все элементы фиксированной дефолтной ширины
**Гипотеза (юзер):** якорь сейчас `variableLength` (-1). Если оба фиксированной ширины
(`NSSquareStatusItemLength` или явная), расстановка детерминированнее.
**Правка:** `status_bar::create` — якорь тоже фиксированной ширины вместо variableLength.
**Риск:** ширина якоря фиксируется — проверить что `<` влезает и кликается.
**Ожидание:** улучшение детерминизма расстановки.

### ИТЕРАЦИЯ 3 — синхронные коллбеки: ждать появления обоих (анти-асинхронность)
**Гипотеза (юзер):** проблема в асинхронности — айтемы появляются не сразу/не вместе.
Пока один не появился (координата валидна), второй не создавать / не действовать.
**Правка:** после создания первого айтема — поллинг его координаты до валидной (≠0),
ТОЛЬКО потом создавать второй. Аналогично перед toggle — ждать оба валидны.
**Замена:** фиксированный STARTUP_HIDE_DELAY=1с → condition-based waiting.
**Ожидание:** устранить остаточные BAD из-за гонки.

### ИТЕРАЦИЯ 4 — ужесточить guard (defense-in-depth)
**Гипотеза:** guard принимает x=0 как «левее». Отвергать невалидное.
**Правка:** `spacer_is_left_of_anchor`:
- отвергать если `spacer.x == 0` или `anchor.x == 0` (не размещён);
- требовать осмысленный зазор (спейсер левее якоря, оба > 0).
**Ожидание:** даже если расстановка сломалась — guard НЕ раздувает на 0, якорь не уезжает.

проверить оба порядка. Финально выбрать тот, что даёт 100% OK.

## Правило применения
- Каждая итерация коммитится отдельно ТОЛЬКО после вердикта «помогло».
- «Сломало/не помогло» → откат итерации (`git checkout`), переход к следующей гипотезе.
- DIAG-инструментарий (poll-таймер) держим до конца всех итераций, убираем перед финальным коммитом.
